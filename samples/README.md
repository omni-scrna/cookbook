# Sample files

**This is intentionally hacky for now.** Until omnibenchmark has a proper "fixture"
mechanism, we commit one real output per stage so contributors can develop and
test a module without re-running the whole benchmark.

See [**INDEX.md**](./INDEX.md) for the current list of sample files with the
producing module and git commit for each. That file is regenerated by
`pixi run samples` (see below).

See [`../README.md`](../README.md) for stage inputs/outputs and module conventions.

## Provenance

Samples here are produced by `scripts/collect-samples.sh`, which picks the
**first file matching each declared output pattern** from a real `out/` tree.
That means:

- Samples are tied to one specific parameter combination — not a "representative"
  sample, just **a** sample.
- Samples can become stale when stages change shape. Re-run the collect script
  whenever a stage's output schema changes.
- File sizes should stay small enough for plain git. If a sample grows past a
  few MB, switch to LFS, fetch-on-demand or the s3 archival backend (open question).

## How to regenerate

Expects `../split-stages-plan` to be checked out alongside this repo with a
fresh `out/` from `ob run`. Then, from the cookbook root:

```bash
pixi run samples
git add samples && git commit -m "refresh samples from <commit>"
```

That runs `scripts/collect-samples.sh` (copies one file per stage from the
benchmark's `out/` tree, writing a `PROVENANCE.txt` next to each sample) and
then `scripts/build-modules-table.py` to regenerate `INDEX.md` with the
module + git commit that produced each sample.
